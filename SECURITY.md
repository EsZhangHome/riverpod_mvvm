# 安全策略

## 支持范围

| 版本 | 安全修复策略 |
| --- | --- |
| `main` | 持续维护，用于下一版本开发 |
| 最新正式 Tag | 发布正式版本后应优先维护 |
| 更早版本 | 默认不承诺修复，请先升级后复现 |

这是一个企业项目底座。复制到真实项目后，项目负责人必须根据发布周期、客户合同和安全等级，更新支持版本、
私有联系方式及响应 SLA，不能直接把模板内容当成公司的正式承诺。

## 私下报告漏洞

以下问题不要创建公开 Issue：

- Token、Cookie、密码、签名密钥或个人信息泄露。
- 登录绕过、越权访问、不安全深链或会话竞态。
- 不可信证书、请求重放、敏感日志或存储明文问题。
- CI Secret、发布产物、依赖供应链或平台权限漏洞。

请使用当前仓库的
[GitHub Security Advisory](https://github.com/EsZhangHome/riverpod_mvvm/security/advisories/new) 私下报告。
如果底座已经复制到公司的私有仓库，应改用该项目自己的安全邮箱、工单或漏洞平台。

报告中请包含：

1. 受影响的 Commit、Tag、平台和构建模式。
2. 最小复现步骤或概念验证，但不要附带真实用户数据。
3. 可能影响的机密性、完整性和可用性。
4. 已知缓解措施，以及是否已经公开披露。

## 处理原则

- 维护者先确认问题并限制信息传播，再评估严重程度和受影响版本。
- 修复分支、补丁和发布时间在公开前保持私密。
- 修复必须包含回归测试；涉及密钥泄露时还必须轮换密钥，不能只删除 Git 历史中的文本。
- 对外披露应在受支持版本已经提供修复之后进行，并向报告者致谢（除非报告者要求匿名）。

## 不属于安全漏洞的内容

- 普通功能建议、UI 调整和无法复现的性能感受。
- 已停止支持版本中、且在最新版本已经修复的问题。
- 只依赖设备已 Root/Jailbreak、攻击者已完全控制系统的理论问题；若仍可突破 App 自身保护，可继续私下报告。

## 接入真实项目时必须修改

- Security Advisory 或私有漏洞平台地址。
- 安全负责人、应急联系人和法务/合规升级路径。
- 严重等级标准、确认时限、修复时限和客户通知机制。
- Android、iOS、后端、Web 和三方 SDK 各自的支持版本。
